Golden rule

ဒီအများလက်ခံထားတဲ့ (သို့) ကိုယ်ပိုင် စည်းမျည်းတွေကို အမြဲသုံးဖို့ အားပေးပါရစေ။ သေးသေးလေးပဲဖြစ်ဖြစ် အကြီးကြီးပဲဖြစ်ဖြစ် အမှားပါခဲ့ရင် အသိပေးဖို့ တောင်းဆိုပါတယ်။ ဒီ guide မှာ ထပ်ထည့်ချင်ရင် သို့ ကူညီချင်ရင် ကျေးဇူးပြုပြီး Github မှာ issue ဖွင့်ပေးပါ။

ပါဝင်တဲ့လူ(contributor) ဘယ်လောက်များများ code လိုင်းကြောင်းတိုင်းက တစ်ယောက်တည်း ရေးထားသလို ဖြစ်နေပါစေ။

Practicality over purity

HTML စံနှုန်းနဲ့ လုပ်ထုံးတွေကို လိုက်နာဖို့ ကြိုးစားပါ။ ဒါပေမဲ့ လက်တွေ့မှာ အသုံးမဝင်တာတွေကိုတော့ လိုက်လုပ်စရာမလိုပါဘူး။ တတ်နိုင်သမျှ markup နည်းနည်းနဲ့ ရှင်းရှင်းလင်းလင်း ရေးပါ။

Attribute order

ဖတ်ရတာလွယ်အောင် HTML attributes တွေကို အစဥ်လိုက် သတ်သတ်မှတ်မှတ် ရေးပါ။

  • class
  • id, name
  • data-*
  • src, for, type, href, value
  • title, alt
  • role, aria-*

Class တွေက ပြန်သုံးလို့ ကောင်းတဲ့ component တွေဖြစ်တဲ့အတွက် သူတို့က အရင်လာရပါမယ်။ Id တွေက အတိအကျသုံးရတဲ့ အရာမို့လို့ လျော့သုံးသင့်ပါတယ် (ဥပမာ၊ in-page bookmarks)။ ဒါကြောင့် သူတို့က ဒုတိယ အနေနဲ့ လာပါမယ်။

Boolean attributes

Boolean attribute ဆိုတာ value သတ်မှတ်ပေးစရာမလိုတဲ့အရာပါ။ XHTML မှာ value ထည့်ပေးဖို့လိုပေမဲ့ HTML5 မှာတော့ မလိုပါဘူး။

ဒီ့ထက် ပိုဖတ်ချင်ရင်တော့ WhatWG ရဲ့ on boolean attributes ဆိုတဲ့ အပိုင်းကို ကြည့်ပါ:

Element တစ်ခုမှာ boolean attribute ပါရင် true value လို့ သတ်မှတ်ပါတယ်။ မပါရင်တော့ false value လို့ မှတ်ယူပါတယ်။

မလိုအပ်ပေမဲ့ attribute value ကို ထည့်မှရမယ်ဆိုရင်တော့ ဒီ WhatWG guideline အတိုင်းလိုက်လုပ်ပါ။

Attribute ထည့်မယ်ဆိုရင် သူ့ value သည် string အလွတ် သို့မဟုတ် [...] attribute ရဲ့ အမည် (အစနဲ့ အဆုံးမှာ မမြင်ရတဲ့ whitespace တွေမပါပဲ) ဖြစ်ရပါမယ်။

တိုတိုပြောရရင် value တစ်ခုမှ မထည့်ပါနဲ့။

CSS

Syntax

  • Environment ပြောင်းသွားလည်း အမြဲပုံစံတူပေါ်အောင် space နှစ်ချက်နဲ့ soft tab ကိုသုံးပါ။
  • Selector တွေကို ပေါင်းရေးရင် တစ်လိုင်းမှာ selector တစ်ခုပဲ ရေးပါ။
  • Block ကြေငြာရင် ဖတ်ရတာ ရှင်းလင်းအောင် ကွင်းစ အရှေ့မှာ space တစ်ခုခံပေးပါ။
  • Block ကွင်းပိတ်ကို လိုင်းအသစ်မှာ ရေးပါ။
  • Declaration တိုင်းမှာ : အဆုံးမှာ space တစ်ခုထည့်ပေးပါ။
  • Error report မှာ တိတိကျကျသိရအောင် declaration တိုင်းကို တစ်လိုင်းစီခွဲပြီး ရေးပေးပါ။
  • Declaration တိုင်းကို semi-colon နဲ့ ပိတ်ပေးပါ။ နောက်ဆုံး declaration ကတော့ ချန်ခဲ့လို့ရပါတယ်။ ဒါပေမဲ့ error ပိုဖြစ်တတ်ပါတယ်။
  • Comma ခံရေးရတဲ့ value တွေမှာ comma တစ်ခုပြီးတိုင်း space တစ်ခုထည့်ပေးပါ။ (ဥပမာ၊ box-shadow)
  • rgb(), rgba(), hsl(), hsla(), သို့ rect() တွေရဲ့ အတွင်းက value တွေမှာတော့ comma နောက်မှာ space မထည့်ပါနဲ့။ ဒီနည်းအားဖြင့် color value တွေနဲ့ တစ်ခြား property value တွေကို ခွဲခြားရလွယ်ကူစေပါတယ်။
  • Color parameter နဲ့ property value တွေကို သုညနဲ့ စ မရေးပါနဲ့။ (ဥပမာ၊ 0.5 အစား .5၊ -0.5px အစား -.5px
  • Hex value အားလုံးကို စာလုံးအသေးနဲ့ ရေးပါ။ ဥပမာ၊ #fff။ စာလုံးအသေးတွေက တစ်ခုနဲ့ တစ်ခု ပုံသဏ္ဌာန်မတူတဲ့အတွက် Document ကို တစ်ချက်ကြည့်ရုံနဲ့ အလွယ်တကူ ခွဲမြင်နိုင်လို့ပါ။
  • ရေးလို့ ရတဲ့နေရာတိုင်း hex value တွေကို အတိုကောက်ရေးပါ။ ဥပမာ၊ #ffffff အစား #fff။
  • Selector ထဲက attribute value တွေကို quote နဲ့ ရေးပေးပါ။ ဥပမာ၊ input[type="text"]။ တစ်ချို့နေရာတွေမှာပဲ ချွင်းချက်ရှိပြီး တသတ်မတ်တည်းဖြစ်ဖို့ ကောင်းတဲ့ အကျင့်တစ်ခုပါ။
  • သုည တန်ဖိုးတွေမှာ unit မထည့်ပါနဲ့။ ဥပမာ၊ margin: 0px; အစား margin: 0;။

ဒီမှာ သုံးသွားတဲ့ အခေါ်အဝေါ်တွေနဲ့ ပတ်သက်ပြီး မေးစရာရှိမယ်ဆိုရင် Wikipedia the Cascading Style Sheets ဆောင်းပါးရဲ့ syntax အပိုင်း ကိုကြည့်ပါ။

  2. Box model

Positioning က ဘာကြောင့် အရင်လာရလည်း ဆိုတော့ element ကို document ရဲ့ ပုံမှန် flow ကနေ ဖယ်ထုတ်ပစ်နိုင်ပြီး box model နဲ့ ဆိုင်တဲ့ style တွေကို ပြင်ရေးလိုက်နိုင်လို့ပဲ ဖြစ်ပါတယ်။ အဲ့ဒါပြီးရင်တော့ component နေရာချပုံနဲ့ dimensions နဲ့ ဆိုင်တဲ့ Box model လာပါမယ်။

Single declarations

Declaration တစ်ခုပဲပါတဲ့ အခါမှာတော့ ဖတ်ရလွယ်၊ ပြင်ရမြန်အောင် rule set မှာ တစ်လိုင်းမဆင်းလို့ရပါတယ်။ တစ်ခုထက်ပိုတဲ့ declaration ဆိုရင်တော့ တစ်လိုင်းစီခွဲပေးပါ။

Shorthand notation

တမင်ကြေငြာဖို့လိုတဲ့ အခါကလွဲပြီး value အားလုံးကို အတိုကောက်ရေးတာကို လျော့ဖို့ ကြိုးစားပါ။ အတိုကောက်အဖြစ်နဲ့ အသုံးလွန်တဲ့ property တွေကတော့

  • padding
  • margin
  • font
  • background
  • border
  • border-radius

တို့ပဲ ဖြစ်ပါတယ်။

အတိုကောက် ဖော်ပြလို့ရတဲ့ value တိုင်းကို မကြာခဏ သတ်မှတ်ပေးစရာမလိုပါဘူး။ ဥပမာအနေနဲ့၊ HTML heading မှာ top နဲ့ bottom margin ပဲ သတ်မှတ်ကြပါတယ်။ ဒါကြောင့် လိုအပ်တဲ့အခါမှသာ အဲ့ဒီ value နှစ်ခုကို ပြင်ရေးပါ။ အတိုကောက် property တွေကို အလွန်အကျွံသုံးခြင်းအားဖြင့် မလိုအပ်ပဲ ပြင်ရေးမိတာ(overwrite)၊ မရည်ရွယ်တဲ့ အကျိုးသက်ရောက်မှုတွေဖြစ်ပြီး အမှားများတဲ့ code ဖြစ်စေတတ်ပါတယ်။

အတိုကောက် ရေးပုံရေးနည်းနဲ့ သူတို့ ဘယ်လိုအလုပ်လုပ်လဲဆိုတာ သိချင်တဲ့ လူတွေအတွက် Mozilla Developer Network မှာshorthand properties ဆိုတဲ့ အရမ်းကောင်းတဲ့ ဆောင်းပါးတစ်ပုဒ်ရှိပါတယ်။

Code ကို ရေးသား ထိန်းသိမ်းပြုပြင်သူများမှာ လူတွေပဲ ဖြစ်ပါတယ်။ အဲ့ဒါကြောင့် မိမိ code ကို သူများ နားလည်လွယ်အောင် comment များနှင့် ရည်ရွယ်ချက်ကို ရှင်းလင်းအောင် ရေးသားပေးပါ။ ကောင်းမွန်တဲ့ code မှာ အကြောင်းအရာနဲ့ ရည်ရွယ်ချက်ကို comment များမှာ ဖော်ပြထားပါတယ်။ Component သို့ class name ကို ဒီအတိုင်းပြန်မရေးပါနဲ့။

Class names

  • Class name တွေကို စကားလုံးအသေးနှင့် dash သုံးပါ။​ (underscore သို့ camelCase မသုံးပါနှင့်)။ ဆက်စပ်နေတဲ့ class တွေကို ပုံမှန်အားဖြင့် dash နဲ့ခွဲပါတယ်။ (ဥပမာ၊​ .btn နှင့် .btn-danger)
  • တတ်နိုင်သလောက် class name တွေကို တိုတိုနှင့် လိုရင်းသတ်မှတ်ပါ။
  • အလွန်အကျွံ နာမည် အတိုကောက်ပေးခြင်းကို ရှောင်ပါ။ button ကို .btn လို့ပေးတာ အသုံးဝင်ပေမဲ့ .s ဆိုတာမျိုးကတော့ ဘာအဓိပ္ပါယ်မှ မသက်ရောက်ပါ။
  • အဓိပ္ပါယ်ရှိတဲ့ နာမည်သုံးပါ။ အမြင်အရပေးတာထက် လုပ်ဆောင်ပုံ၊ တည်ဆောက်ပုံအရ ပေးပါ။
  • အနီးဆုံး parent သို့ base class နာမည်ပေါ် မူတည်ပြီး class နာမည် အစ (prefix) ကို သတ်မှတ်ပါ။
  • (style နှင့် ဆန့်ကျင်ဖက်ဖြစ်သော)လုပ်ဆောင်ပုံကို သတ်မှတ်ရာတွင် class များကို .js-* နှင့်စပေးပါ။ သို့သော် ထို​ class များကို CSS မှာ ထည့်မရေးပါနှင့်။

Sass နှင့် Less variable အမည်များ သတ်မှတ်ရာတွင်လည်း ဤစည်းမျဥ်းများကို အသုံးပြုနိုင်ပါသည်။

Selectors

  • Rendering performance အကောင်းဆုံးဖြစ်အောင် element tag အစား class ကို သုံးပါ။
  • အမြဲသုံးတဲ့ component တွေကို attribute selectors အများအပြားသုံးခြင်းကို ရှောင်ပါ။ (ဥပမာ၊​ [class^="...")။ ထိုသို့လုပ်ခြင်းသည် browser စွမ်းဆောင်ရည်ကို ထိခိုက်စေတယ်လို့ သိထားပါတယ်။
  • Selector များကို တိုအောင်ထားပါ။ Selector တစ်ခုမှာ element သုံးခု အထိပဲ ကြိုးစား သတ်မှတ်ပေးပါ။
  • လိုအပ်ရင် class ကို အနီးဆုံး parent scope တစ်ခုထဲမှာပဲထားပေးပါ။ (ဥပမာ၊ prefixed class တွေ မသုံးတဲ့ အခါ)
